# 02-current-task.md

## 이번 작업 목표

현재 레포지토리에 Vite + React + Tailwind CSS 기반 프론트엔드 프로젝트를 생성하고, 맛길 MVP 화면 껍데기를 구현한다.

현재 목표는 실제 서비스 완성이 아니라, **작동하는 화면 흐름을 먼저 만드는 것**이다.

## 작업 순서

1. 현재 레포지토리 상태를 확인한다.
2. Vite + React + JavaScript 프로젝트를 생성한다.
3. Tailwind CSS를 설정한다.
4. `01-implementation-rules.md`의 폴더 구조를 기준으로 폴더와 파일을 만든다.
5. mock 데이터와 mock service 함수를 만든다.
6. 화면들을 구현한다.
7. 하단 메뉴바를 구현한다.
8. 화면 전환이 자연스럽게 작동하는지 확인한다.
9. `npm run build` 또는 가능한 검증 명령으로 오류를 확인한다.
10. 마지막에 사용자가 실행할 명령어를 알려준다.

## 반드시 구현할 화면

### 1. 홈 화면

경로 예시:

```txt
/
```

구현 내용:

* 서비스명 `맛길`
* 설명 문구: `서울에서 외국인 관광객을 위한 맛집 동선을 추천합니다.`
* 추천 시작하기 버튼
* 하단 메뉴바 표시

조건:

* 첫 화면은 홈 화면이다.
* 사용자는 로그인하지 않아도 추천 시작하기 버튼을 누를 수 있다.

### 2. 로그인 화면

경로 예시:

```txt
/login
```

구현 내용:

* 서비스명 `맛길`
* 이메일 입력창
* 비밀번호 입력창
* 이메일 로그인 버튼
* 이메일 회원가입 버튼
* Google로 계속하기 버튼
* Facebook으로 계속하기 버튼
* 로그인 없이 둘러보기 버튼

조건:

* 실제 로그인 백엔드는 구현하지 않는다.
* 버튼 클릭 시 mock 로그인 상태 또는 게스트 상태로 홈 화면으로 이동한다.
* Google, Facebook 로그인은 UI만 만든다.
* 외부 로고 이미지나 무거운 아이콘 라이브러리는 무리하게 추가하지 않는다.

### 3. 지역/랜드마크 선택 화면

경로 예시:

```txt
/areas
```

구현 내용:

선택 가능한 지역 버튼:

* 홍대
* 성수
* 강남
* 종로
* 명동
* 경복궁

추가 UI:

* 검색 입력창
* MVP는 서울 지역만 지원한다는 안내 문구
* 다음 버튼

조건:

* 검색 입력창은 UI만 제공해도 된다.
* 선택한 지역은 추천 흐름에서 사용할 수 있게 상태로 관리한다.

### 4. 취향 선택 화면

경로 예시:

```txt
/preferences
```

구현 내용:

취향 선택은 필수 입력이 아니다.

작은 안내 문구를 표시한다.

```txt
로그인하면 취향을 저장할 수 있습니다.
```

음식 종류:

* 한식
* 일식
* 중식
* 양식
* 분식

예산:

* 저렴함
* 보통
* 상관없음

버튼:

* 건너뛰기
* 추천 받기

조건:

* 취향을 선택하지 않아도 기본 추천으로 진행한다.
* 선택한 취향은 mock 추천 흐름에 전달한다.

### 5. 추천 생성 중 화면

경로 예시:

```txt
/loading
```

구현 내용:

* `맛길이 동선을 추천하고 있어요` 같은 로딩 문구 표시
* 1초 정도 후 추천 결과 화면으로 이동

조건:

* 실제 API 호출은 하지 않는다.
* `recommendationService.js`의 mock 함수를 사용한다.
* 추후 Supabase Edge Functions + Solar API 호출로 교체하기 쉽게 작성한다.

### 6. 추천 결과 카드 화면

경로 예시:

```txt
/result
```

구현 내용:

맛집 2곳 + 카페 1곳을 카드로 표시한다.

각 카드 표시 정보:

* 방문 순서
* 장소명
* 카테고리
* 예상 도보 시간
* 추천 이유

거리 기준 안내 문구:

```txt
첫 번째 추천 식당 기준 도보 10분 이내 장소를 우선 추천하고, 부족할 경우 15분 이내까지 확장합니다.
```

버튼:

* 다시 추천 받기
* 자주 쓰는 표현 보기

조건:

* 지도는 넣지 않는다.
* 지도 영역이 필요하면 `지도 시각화는 고도화 예정` 안내 박스만 넣는다.
* 다시 추천 받기 버튼은 mock 추천 결과를 다시 보여주거나 추천 생성 중 화면으로 이동하게 한다.

### 7. 자주 쓰는 표현 화면

경로 예시:

```txt
/phrases
```

구현 내용:

카테고리:

* 입장
* 주문
* 요청
* 결제
* 포장

표현 예시:

입장:

* `두 명이에요.` / `Table for two, please.`
* `자리 있나요?` / `Do you have a table?`

주문:

* `이거 하나 주세요.` / `One of this, please.`
* `추천 메뉴가 있나요?` / `Do you have a recommended menu?`

요청:

* `물 주세요.` / `Water, please.`
* `덜 맵게 해주세요.` / `Please make it less spicy.`

결제:

* `계산할게요.` / `I would like to pay.`
* `카드 돼요?` / `Can I pay by card?`

포장:

* `포장해 주세요.` / `Please pack this to go.`

조건:

* 각 표현마다 한국어 문장과 영어 의미를 표시한다.
* 재생 버튼 클릭 시 브라우저 Web Speech API를 사용해 한국어 TTS를 재생한다.
* TTS가 지원되지 않는 환경이면 alert 또는 안내 문구를 표시한다.

## 추천 로직 mock 처리

현재는 실제 거리 계산을 하지 않는다.

그러나 `recommendationService.js` 내부에 향후 교체 가능한 함수 구조를 만든다.

향후 실제 로직 주석:

```txt
1. TourAPI 또는 Supabase에서 장소 후보 조회
2. 첫 번째 추천 식당 기준 도보 10분 이내 후보 우선 필터링
3. 후보 부족 시 도보 15분까지 완화
4. Solar API가 후보 중 최종 장소 선택과 추천 이유 생성
```

## 화면 진입 조건

첫 화면은 홈 화면으로 한다.

로그인 화면은 별도 경로로 제공한다.

사용자는 로그인하지 않고도 아래 기능을 사용할 수 있어야 한다.

* 지역 선택
* 취향 선택
* 추천 결과 확인
* 자주 쓰는 표현 TTS

북마크와 마이페이지는 로그인 필요 안내 화면으로 처리한다.

## 완료 기준

아래 조건을 만족하면 이번 작업은 완료로 본다.

1. `npm run dev`로 화면을 확인할 수 있다.
2. 첫 화면은 홈 화면이다.
3. 로그인하지 않아도 추천 흐름을 진행할 수 있다.
4. 지역 선택 → 취향 선택 → 로딩 → 추천 결과 흐름이 작동한다.
5. 추천 결과는 맛집 2곳 + 카페 1곳 카드로 표시된다.
6. 자주 쓰는 표현 화면에서 한국어 문장과 영어 의미가 표시된다.
7. TTS 재생 버튼이 동작하거나, 미지원 환경 안내가 나온다.
8. 하단 메뉴바의 5개 메뉴가 각 화면으로 이동한다.
9. 북마크와 마이페이지는 로그인 필요 안내 화면으로 표시된다.
10. 프론트엔드에 `.ts`, `.tsx` 파일이 없다.
11. 가능한 경우 `npm run build` 검증이 통과한다.
